# 后续优化与维护建议

> 基于 Demo v1.0 开发过程中积累的经验，对后续迭代的实用建议。

---

## 一、架构优化（优先级：高）

### 1.1 拆分 App.tsx

当前 `App.tsx` 182KB 是项目最大的单文件，承载全部流程编排、flag 分支、对话触发、地图切换、NPC 事件、演出层调度。随着后续章节和演出类型增加，风险会持续升高。

**建议**：

- 抽离 `GameFlowController` 模块：独立管理整个游戏流程（场景切换、flag 状态机、触发分发）
- 抽离 `PerformanceLayerManager` 模块：统一调度 CG / 对话 / 手机群聊 / 屏幕文字等演出层的切换和生命周期
- 抽离 `NpcEventSystem` 模块：将 NPC 生成、移动、表情切换、对话触发等逻辑从 App.tsx 剥离
- 目标：将 App.tsx 缩减至 50KB 以内，作为顶层编排器而非逻辑承载者

### 1.2 统一演出层调度

当前 CG、对话、手机群聊、屏幕文字、特效五种演出模式各自管理生命周期，容易出现状态冲突（如对话框和 CG 同时显示）。

**建议**：

- 定义统一的 `PerformanceLayer` 接口，每个演出层实现 `show()` / `hide()` / `isActive()`
- 建立 `PerformanceQueue`：按优先级排队，确保上一演出完成后再启动下一演出
- CG 层和对话层的定位规则统一为配置驱动（当前 `.cg-choices` 的 `bottom: 14%` 是硬编码）

### 1.3 前后端类型统一

当前 `client/types/game.ts` 和 `server/types/game.ts` 独立维护，已出现字段不一致的风险。

**建议**：

- 创建 `shared/types/` 目录，使用 TypeScript `paths` 别名共享核心类型
- 或者将 server 端类型改为从 client 端导入（通过 monorepo 的 workspace 引用）

## 二、剧本数据管线（优先级：高）

### 2.1 自动校验脚本

当前 `scenes.ts`（345KB / 240+ 节点）和 `ch2Scenes.ts` 纯手工维护，出错全靠手动跑游戏发现。

**建议**：编写 `scripts/validate-scenes.ts`，实现：

- 检查所有 `nextSceneId` 是否指向存在的节点
- 检查所有 `choice.nextSceneId` 是否指向存在的节点
- 检查所有角色码（`[NPC:xxx]`、`[主角]`、`[旁白]`）是否使用了已定义的 speaker 集合
- 检查 `\n\n` 段落分隔符是否被意外删除（搜索 AI 生成附近的节点）
- 检查是否所有 `aiEvent` 节点都有对应的 `aiSceneConfig` 配置
- 集成到 `npm run validate` 或 pre-commit hook

### 2.2 AI 场景配置一致性检查

`aiSceneConfig.ts`（65KB / 22 个场景）中的 `skeletonLines`、`requiredLines`、`dynamicBranches` 和 `scenes.ts` 中的 prompt 文本需要一一对应。

**建议**：

- 编写校验：每个 `aiSceneConfig` key 是否都在 `scenes.ts` 中有引用
- 编写校验：`fallback` 文本的行数是否在 `min_lines`-`max_lines` 范围内
- 编写校验：`dynamicBranches` 中引用的 trait（如 `trust`、`empathy`）是否在七维定义中存在

### 2.3 角色名拼写词典

已出现过 `defult` vs `default` 的拼写问题。角色码、立绘文件名、NPC ID 应统一维护。

**建议**：

- 创建 `client/src/data/characterRegistry.ts`：定义所有角色的 ID、中文名、立绘路径、portrait 文件名
- `scenes.ts` 解析角色码时从 registry 查询，拼写错误在编译/校验期暴露
- 举例格式：

```ts
export const CHARACTERS = {
  liuyu: { name: '刘宇', portraitDir: 'portraits/ly_', expressions: ['smile', 'sad', 'angry'] },
  wangTeacher: { name: '王沁林', portraitDir: 'portraits/wang_', expressions: ['default'] },
  // ...
};
```

## 三、AI 系统（优先级：中）

### 3.1 Prompt 可调试性

当前 AI 生成的 prompt 是在服务端组装后直接发送，调试时难以复现问题。

**建议**：

- 在开发模式下记录每次 AI 调用的完整 prompt 和 response 到日志文件
- 增加 `/api/debug/scene-prompt?sceneId=xxx` 调试接口，返回组装后的完整 prompt（不实际调用 LLM）
- 在 mock 模式下，同时输出"如果真实调用，prompt 会是什么"供人工审查

### 3.2 Fallback 热更新

当前 fallback 文本硬编码在 `aiSceneConfig.ts` 中，修改需要重新编译。

**建议**：

- 将 fallback 文本从 TypeScript 代码中抽离到独立的 JSON/Markdown 文件
- 开发模式下支持热加载 fallback 文件，方便调校
- 生产构建时内联回代码中保持单文件部署

### 3.3 上下文窗口管理

随章节增加，`aiMemory.ts` 维护的记忆体量持续膨胀（主角画像 + 世界观 + 所有 NPC 印象 + 重要事件），可能超出模型上下文窗口。

**建议**：

- 实现上下文优先级裁剪：当前场景相关的 NPC 记忆保留全部，无关 NPC 只保留摘要
- 事件历史按重要性分层：关键选择保留原文，次要事件只保留语义标签
- 定期合并同类事件：如"多次帮助周隽秀"合并为综合印象而非逐条罗列

## 四、存档系统鲁棒性（优先级：高）

### 4.1 存档版本号与迁移

当前存档数据结构没有版本号，未来修改 GameState 结构会导致旧存档不可用。

**建议**：

- 在存档中增加 `saveVersion: number` 字段
- 实现 `migrateSave(save, fromVersion, toVersion)` 迁移函数
- 加载存档时自动检测版本号，按需迁移

### 4.2 存档完整性校验

读档时如果某字段缺失（如新增 trait 维度），会导致 undefined 传播。

**建议**：

- 存档时写入完整 GameState 快照（含所有默认值，不省略零值字段）
- 读档时用 `GameState` 默认值做 `deepMerge`，缺失字段自动填充默认值
- 读档后自动触发一次运行时状态校验（地图是否存在、NPC 配置是否有效）

### 4.3 存档恢复检查清单

每次新增功能后，检查是否遗漏存档维度：

| 维度 | 当前状态 |
|------|----------|
| 地图 ID + 玩家坐标 | ✅ |
| NPC 位置和状态 | ✅ |
| 当前对话/场景节点 | ✅ |
| flag 状态机 | ✅ |
| BGM 播放状态 | ⚠ 需验证 |
| AI 记忆结构 | ⚠ 需验证 |
| CG 层状态 | ⚠ 需验证 |
| 手机群聊进度 | ❌ 未处理 |

## 五、音频系统（优先级：中）

### 5.1 音频状态机

当前 BGM 开始/结束、音效绑定台词、CG 无图继承地图 BGM 等逻辑分散在多处。

**建议**：

- 抽象 `AudioStateMachine`：接收"场景切换""对话开始""对话结束""CG 进入""CG 退出"等事件，内部计算正确的 BGM 行为
- CG 无图时继承地图 BGM 的逻辑改为状态机的默认行为，而非在每个分支中硬编码

### 5.2 音效绑定台词

当前音效从"定时播放"改为"绑定台词"后，需要精确匹配台词在对话序列中的位置。

**建议**：

- 在 Scene 数据中增加 `sfx?: string` 字段，绑定到该节点台词播放完毕时触发
- 避免通过 `sceneId` 含有特定前缀（如 `dorm_act3_`）来隐式推断播放音效

## 六、地图与演出（优先级：中）

### 6.1 地图编辑器增强

`map_editor.html` 当前可编辑 tile、碰撞体和 spawn 点，但缺少数 NPC 路径编辑。

**建议**：

- 在地图编辑器中增加 NPC 路径编辑模式：拖拽路径点，导出为 `npc_paths` 配置
- 在编辑器中预览 trigger 触发范围（半透明色块）
- 导出时自动校验 spawn 点 ID 不与已有冲突

### 6.2 演出动作库

当前地图演出动作（NPC 移动、镜头平移、冻结玩家、特效）都是手写事件字符串。

**建议**：

- 定义 `MapAction` 类型系统：`{ type: 'move_npc' | 'camera_pan' | 'freeze' | 'sfx' | ..., params: ... }`
- `scenes.ts` 中通过 `actions: MapAction[]` 数组声明演出序列，而非手写事件名
- App.tsx 中的演出调度器按数组顺序逐个执行，完成后自动触发 `nextSceneId`

## 七、内容补全（优先级：中—长期）

### 7.1 暂缺章节

当前 Demo 缺第二、三、七章。补充时注意：

- **第二章（副本开启）**：已独立为 `ch2Scenes.ts`，部分骨架存在。需补充完整的家庭探索流程
- **第三章（学校初入）**：教室、空座位谜题、刘宇介入。地图 `classroom` 已就绪，需补充场景数据
- **第七章（家庭真相）**：车内、客厅对话、父亲失业。地图 `livingroom` / `balcony` 已就绪

### 7.2 Demo 后的扩展方向

按优先级排序：

1. **违规压力 UI**：屏幕噪声、心跳、呼吸效果——视觉冲击力强，比赛展示效果好
2. **镜中空间**：家庭线核心视觉，需要滤镜/后处理
3. **试胆活动**（第 9 章+）：湖中遗物、书中落叶——新地图 + 新谜题
4. **结局池扩展**：救世主幻觉结局需要后续章节支撑

## 八、测试与质量保障（优先级：中—长期）

### 8.1 关键路径冒烟测试

不需完整单元测试，但核心流程应有自动化冒烟。

**建议**：

- 编写 `scripts/smoke-test.ts`：无头加载 → 点击"开始游戏" → 推进到第一个选择 → 验证 trait 变化 → 退出
- 集成到 CI（如果后续配置 GitHub Actions）

### 8.2 资源完整性检查

**建议**：

- 检查 `scenes.ts` 中引用的所有 NPC name 是否在 `characters.ts` 中定义
- 检查 `scenes.ts` 中引用的所有 portrait 文件名是否在磁盘上存在
- 检查所有地图的 `map.json` 引用的 tileset 图片是否存在

## 九、部署与运维（优先级：低—需要时）

### 9.1 一键部署脚本

**建议**：

- 编写 `deploy.sh`：安装依赖 → 构建前端 → 构建后端 → 启动服务
- 支持 Docker 部署（`Dockerfile` + `docker-compose.yml`）

### 9.2 环境配置文档化

当前 `MODEL_PROVIDER`、`OPENAI_API_KEY`、`LLM_PROXY_URL` 等环境变量缺少文档。

**建议**：

- 在 README 中增加"环境变量配置"章节
- 提供 `.env.example` 模板文件

---

## 十、文档维护纪律（优先级：高—持续）

从本次开发中吸取的最大教训：**文档脱节会显著增加后续开发成本**。

**建议**：

- 每次新增/修改功能后，同步更新对应文档（推荐在 commit 前完成）
- 在地图新增/修改后，更新 `docs/01_像素美术风格规范.md` 中的地图清单
- 在新增 NPC 或修改 AI 配置后，更新 `docs/快乐小孩_AI叙事游戏项目总览_v0.1.md` 中的角色卡
- 建议在 `README.md` 末尾维护一个"最新变更"小节，记录最近 5 次关键变更

## 优先级总览

| 优先级 | 事项 |
|--------|------|
| 🔴 高 | 拆分 App.tsx、统一演出层、自动校验脚本、存档版本号与迁移 |
| 🟡 中 | Prompt 调试、Fallback 热加载、音频状态机、地图编辑器增强、演出动作库 |
| 🟢 中-长期 | 内容补全（二/三/七章）、冒烟测试、资源完整性检查 |
| 🔵 低 | 部署脚本、环境变量文档化 |
| ⚪ 持续 | 文档同步维护 |
